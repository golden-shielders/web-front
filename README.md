## 1. 프로젝트 개요

게시판 서비스의 프론트엔드 애플리케이션이다.  
React Router 기반 SPA 구조로 로그인, 게시글 목록 조회, 게시글 상세 조회, 게시글 작성 기능을 제공한다.  
인증이 필요한 페이지는 보호 라우트(Protected Route)를 통해 접근을 제한한다.

| 항목        | 내용                                       |
| ----------- | ------------------------------------------ |
| 프로젝트명  | golden_shielders                           |
| 앱 형태     | SPA (Single Page Application)              |
| 주요 기능   | 로그인, 게시글 목록/상세 조회, 게시글 작성 |
| 인증 방식   | Access Token 기반 인증                     |
| 보호 페이지 | `/write`                                   |

## 4. 환경 설정

## 5. 라우팅 구조

`createBrowserRouter` 기반으로 전체 페이지를 구성한다.

| 경로             | 설명             | 접근 권한   |
| ---------------- | ---------------- | ----------- |
| `/`              | 홈 화면          | 공개        |
| `/login`         | 로그인 화면      | 공개        |
| `/posts`         | 게시글 목록      | 공개        |
| `/posts/:postId` | 게시글 상세      | 공개        |
| `/write`         | 게시글 작성      | 로그인 필요 |
| `*`              | Not Found 페이지 | 공개        |

### 보호 라우트 동작

- 인증 확인 중이면 `인증 확인 중...` 표시
- 비로그인 상태이면 `/login`으로 리다이렉트
- 원래 접근하려던 경로를 `state.from`에 저장

## 6. 인증 및 상태 관리

인증 상태는 `AuthContext`에서 전역 관리한다.

### 인증 흐름

| 단계           | 설명                                                  |
| -------------- | ----------------------------------------------------- |
| 1. 초기 로딩   | `getValidUserFromStorage()`로 저장된 사용자 정보 복원 |
| 2. 로그인 요청 | `loginApi(form)` 호출                                 |
| 3. 토큰 저장   | `saveAccessToken(data.accessToken)` 수행              |
| 4. 사용자 추출 | `getUserFromToken()`으로 사용자 정보 파싱             |
| 5. 상태 반영   | `setUser(parsedUser)`로 로그인 상태 반영              |
| 6. 로그아웃    | `clearAccessToken()` 후 사용자 상태 초기화            |

### AuthContext 제공 값

| 항목              | 설명                        |
| ----------------- | --------------------------- |
| `user`            | 현재 로그인 사용자 정보     |
| `isLoading`       | 초기 인증 상태 확인 중 여부 |
| `isAuthenticated` | 로그인 여부                 |
| `login`           | 로그인 수행 함수            |
| `logout`          | 로그아웃 수행 함수          |

### useAuth

`useAuth` 훅은 `AuthContext`를 쉽게 사용하기 위한 커스텀 훅이다.  
Provider 바깥에서 사용할 경우 에러를 발생시킨다.

## 7. 주요 화면 기능

### 7-1. 글쓰기 페이지

`WritePage`는 제목, 내용, 첨부파일을 입력받아 게시글을 생성하는 페이지이다.

| 기능             | 설명                             |
| ---------------- | -------------------------------- |
| 제목 입력        | `input[name="title"]`            |
| 내용 입력        | `textarea[name="content"]`       |
| 파일 첨부        | `input type="file" multiple`     |
| 업로드 목록 표시 | 선택한 파일명과 크기 표시        |
| 제출 처리        | 등록 중 상태 표시 및 에러 처리   |
| 성공 후 이동     | 생성된 게시글 상세 페이지로 이동 |

### 7-2. 보호 페이지 접근 제어

- `/write` 경로는 로그인 사용자만 접근 가능
- 인증 실패 시 로그인 페이지로 이동

## 12. 알려진 취약점

| 번호 | 위치                 | 취약점                                                | 위험도  |
| ---- | -------------------- | ----------------------------------------------------- | ------- |
| 1    | `activateScripts.ts` | 삽입된 `<script>` 태그를 재실행하여 XSS 위험 증가     | 🔴 높음 |
| 2    | 토큰 저장 구조       | XSS 발생 시 Access Token 탈취 위험 가능               | 🟠 중간 |
| 3    | ProtectedRoute       | 클라이언트 측 접근 제한만으로는 완전한 인가 보장 불가 | 🟠 중간 |

### 상세 설명

#### 12-1. script 재실행 유틸

`activateScripts(container)`는 컨테이너 내부의 `<script>` 태그를 다시 생성하여 실행시키는 로직이다.  
신뢰되지 않은 HTML을 DOM에 삽입하는 코드와 함께 사용될 경우 XSS를 실제 실행 가능하게 만들 수 있다.

#### 12-2. 토큰 저장 구조 주의

Access Token을 클라이언트 저장소에 보관하는 구조는 XSS가 발생했을 때 토큰 탈취로 이어질 수 있다.

#### 12-3. 보호 라우트 한계

프론트엔드의 보호 라우트는 UX 차원의 접근 제어일 뿐이며, 실제 권한 검사는 반드시 백엔드에서 수행해야 한다.
